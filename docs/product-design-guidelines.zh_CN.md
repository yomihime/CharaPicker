# 产品与设计规范（zh_CN）

本文承载 CharaPicker 的长期产品语义、界面规范和洞察流设计约束。它不是实现进度表；具体功能是否已经完成，应以当前代码、`README.md` 和架构文档为准。

## 1. 产品定位

CharaPicker（拾卡姬）是一个面向二次元素材分析的桌面工具，用于从番剧、漫画、视频、图片或文本素材中提取角色相关信息，并生成结构化角色档案、角色卡或人设资料。

长期产品原则：

- Extract Once：素材尽量只处理一次，沉淀为可复用知识库。
- Targeted Insight：在通用提取结果之上，围绕用户指定的目标角色或世界观输出定向洞察。
- Iterative Compilation：生成角色卡时应逐步更新角色状态，保留成长、冲突和关系变化。
- Visible Thinking：用户应能看到关键洞察流，而不是只看到普通调试日志。

## 2. 产品语气

“拾卡姬”可以有轻微拟人化语气，风格可爱、活泼、二次元，但不能牺牲信息清晰度。

文案原则：

- 用户应明确知道系统正在做什么、为什么等待、失败原因是什么。
- 可爱的表达只能作为辅助，不能遮蔽状态、路径、错误或下一步动作。
- 错误、权限、清理素材、API Key、模型费用等高风险场景优先直白表达。
- UI 可见文案必须通过 `i18n/` 管理，不要长期硬编码在页面代码中。

示例风格：

- 启动：`拾卡姬 (CharaPicker) v0.2.0-alpha.1 已就位`
- 引导：`准备好帮您从番剧/漫画里扒角色卡啦！请投喂视频文件或漫画文件夹吧。`
- 处理中：`拾卡姬正在逐帧观察表情...`
- 格式不支持：`这个格式暂时读不懂，请换成 MP4 或 PNG 试试。`

示例不是固定文案库；实际文案以 `i18n/*.json` 中的键值为准。

## 3. UI、i18n 与资源规范

界面技术与风格：

- 当前应用使用 PyQt6 + qfluentwidgets。
- 优先使用 qfluentwidgets 社区版已有组件，不为常规控件重复造轮子。
- 视觉方向为 Windows 11 Fluent Design / Acrylic，并支持亮色、暗色或跟随系统主题。

多语言：

- 用户可见文本必须放入 `i18n/`，通过 `utils.i18n.t()` 获取。
- 当前基础语系为 `zh_CN`、`zh_TW`、`en_US`、`ja_JP`。
- 新增可见文案时，四个语系文件应同步补键。
- key 应稳定、可读，不要复用语义不同的字符串。

资源与颜色：

- 运行时通用资源放在 `res/`。
- UI 颜色标识先定义在 `res/colors.py`，再由界面代码引用。
- 页面、组件和样式代码中不要直接散落 RGB、RGBA 或 Hex 颜色值。
- 图片、图标、贴图等运行时素材放在 `res/` 下，文件名应语义清楚。
- `res/` 不放用户可见文案；用户可见文本仍归 `i18n/`。

## 4. InsightStreamPanel 规范

`InsightStreamPanel` 是核心 UI 组件，用于展示用户真正关心的结构化洞察。

职责：

- 展示 AI 在提取或编译过程中发现的关键角色信息。
- 展示事件应来自结构化 `dict`，通过 Qt Signal 流入 UI。
- 适合呈现目标角色、片段、关系变化、冲突、成长信号和简短洞察摘要。

不职责：

- 不展示普通调试日志。
- 不作为日志文件浏览器。
- 不把模型底层请求细节、API Key、完整隐私文本或大型原始素材内容展示给用户。

当前实现采用 Card + Timeline 的流式卡片形式，并支持按 `stream_id` 更新同一条流式事件。自动滚动、关键词高亮、按角色过滤等属于设计方向；实现前必须先核对当前代码。

洞察文案应短而明确，例如：

- `第 3 集捕捉到 [角色名] 的微表情变化...`
- `正在调和性格矛盾...`
- `[角色A] 初期表现顺从 -> 第 5 章出现反抗倾向。`

## 5. 混合媒体与成本控制

素材支持方向：

- 支持视频、漫画/图片文件夹、图像和文本素材的混合输入。
- 素材应先进入项目目录，再由处理流程准备为当前可消费的 `materials/`。
- `raw/` 保存可重新处理的源副本；`materials/` 保存当前处理管线实际读取的素材入口。
- 清理 `raw/` 前必须确认 `materials/` 中已有可用素材，并在项目配置中记录清理状态。

成本与预览：

- 正式长流程前应优先提供 Preview 试算。
- Preview 默认只处理有限 chunk；当前代码中的预览上限为 2 个 chunk。
- Preview 目标是尽早暴露模型可用性、预计耗时、费用风险和首段洞察质量。
- 本地模式可以按显存预算反推切片策略；云端模式可以按 token 或费用预算反推切片策略。
- 云端视频预览中的“输出 Token / 分钟”应按视频切片时长换算为单次请求的输出上限，例如 1024/分钟对应 1 分钟约 1024、2 分钟约 2048、30 秒约 512。
- 低输出预算是有效的试错参数，不应硬性阻断预览。低于建议值时先弹窗说明风险并请求确认；用户确认后不要在洞察流重复展示同一条低预算提示。
- 只有模型实际因为输出长度上限截断结构化结果时，才应在洞察流提示该 chunk 被跳过，并继续使用其他可用结果。

这些是产品与流程约束，不代表所有能力都已完整实现。涉及真实素材接入时，应同时查阅 `docs/extraction-workflow.zh_CN.md`、`docs/extraction-development-roadmap.zh_CN.md` 和当前 `core/extractor.py`。

## 6. Prompt 与结构化输出原则

Prompt 正文是可维护资源，不应硬编码在业务代码中。默认 prompt 应维护在 `res/default_prompts.json`，由 `utils.ai_model_middleware` 统一加载、套用用户覆盖并渲染变量。

提取类 prompt 应长期满足：

- 初步提取只面向素材本身，不接收、使用或推断用户预设目标角色列表，避免角色混淆并保证结果可复用。
- 一次调用应保留素材中出现的全部有效角色信息，避免为每个角色重复跑模型。
- 只根据输入素材和结构化上下文输出，不编造输入中不存在的剧情或设定。
- 当前 chunk 新事实优先于历史摘要。
- 输出结构化 JSON。
- 洞察摘要应简短，适合进入 InsightStreamPanel。

编译类 prompt 应长期满足：

- 输入按时间顺序排列。
- 更新而不是重写 `CharacterState`。
- 显式记录冲突、伪装、成长和关系变化。
- 保留证据来源。
- 输出结构化 JSON。
