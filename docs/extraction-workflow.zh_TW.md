# 提取工作流程技術說明（zh_TW）

本文面向使用者、研究者，以及想理解 CharaPicker 設計的人，說明長篇影片素材如何被提取、壓縮、組織，並最終用於生成角色卡。

## 1. 設計目標

CharaPicker 的核心原則是 `Extract Once`：原始影片、字幕或影像素材只解析一次，之後沉澱為可重複使用的結構化知識庫。

這套流程要解決三個問題：

- 長篇番劇或影片素材太長，不能一次性全部放進模型。
- 角色成長需要時間順序，不能只做孤立片段總結。
- 後續生成角色卡時，應優先讀取結構化結果，而不是重複分析原影片。

因此，系統把素材拆成「季、集、chunk」三層：

- `chunk` 是提取階段的處理單位，用來控制模型上下文長度。
- `集` 是劇情理解和角色成長的最小自然單位。
- `季` 是階段性成長、關係變化和長期衝突的總結單位。

## 2. 輸入素材約定

第一階段採用簡單、可解釋的目錄識別規則：

- 使用者選擇一個素材根目錄。
- 根目錄下每個一級資料夾表示一季。
- 每季資料夾中的影片檔案表示該季的集。
- 季資料夾和集檔案預設按名稱排序。

建議命名方式：

- 季資料夾：`01 To LOVEる`、`02 Motto To LOVEる`、`03 Darkness`
- 集檔案：`01 xxx.mp4`、`02 xxx.mp4`、`10 xxx.mp4`
- 也可以使用 `xxx S01`、`xxx S02`，但需要保證名稱排序後符合實際順序。

建議使用補零編號，例如 `01`、`02`、`10`。這樣簡單文字排序就能得到正確順序。

匯入後，系統會生成 `source_manifest.json`，記錄原始資料夾、原始檔案名稱和內部編號的映射。後續流程使用穩定編號，例如 `season_001`、`episode_001`、`chunk_0001`，不會反覆依賴原始檔案名稱推斷。

## 3. 總體流程

推薦流程如下：

```text
原素材
-> 按資料夾識別季
-> 按檔案名稱排序識別集
-> 單集切分為 chunk
-> 提取每個 chunk 的結構化結果
-> 合併為集級完整內容
-> 生成集級壓縮摘要
-> 合併為季級完整內容
-> 生成季級壓縮摘要
-> 按季、按集逐步編譯角色狀態
-> 生成最終角色卡
```

這裡有一個重要設計：角色卡生成不從 chunk 開始。

chunk 只是為了讓模型能處理長素材。真正模擬角色成長時，應從「每集完整內容」開始，按集推進。單集比 chunk 更符合劇情結構，也更適合作為角色變化的觀察單位。

## 4. 提取時的上下文

提取當前 chunk 時，系統按以下優先級組織上下文：

1. 當前 chunk 內容。
2. 當前集已完成 chunk 的完整結構化提取結果。
3. 當前季已完成集的集級壓縮摘要。
4. 前一季的季級壓縮摘要。

其中，當前 chunk 永遠是最高優先級證據。

同一集通常不會太長，所以當前集內已經提取過的 chunk 可以帶「完整結構化結果」，而不是只帶短摘要。但這裡的「完整」指結構化提取結果，不是原始字幕或原文全文。這樣可以保留細節，同時避免重複消耗上下文。

如果某一集非常長，系統可以保留最近幾個 chunk 的完整結構化結果，並把更早 chunk 合併成集內滾動摘要。

## 5. 跨集與跨季

同一季內，後續集會帶上前面已完成集的壓縮摘要。這樣模型能理解角色關係、衝突和狀態變化的連續性。

跨季時，可以帶上前一季的季級壓縮摘要，但它只作為低優先級背景。它負責說明角色進入當前季前的狀態、關係和未解決衝突，不能覆蓋當前季素材中的新事實。

建議語義上把前一季摘要標記為：

```text
PREVIOUS_SEASON_BACKGROUND
```

也就是說，前一季資訊是背景，不是當前證據。

## 6. 知識庫結構

提取結果會寫入專案的 `knowledge_base`，並按季、集、chunk 分層保存。

推薦結構：

```text
knowledge_base/
├── source_manifest.json
├── seasons/
│   ├── season_001/
│   │   ├── season_content.json
│   │   ├── season_summary.json
│   │   ├── character_stage_states.json
│   │   └── episodes/
│   │       ├── episode_001/
│   │       │   ├── episode_content.json
│   │       │   ├── episode_summary.json
│   │       │   └── chunks/
│   │       │       ├── chunk_0001.json
│   │       │       └── chunk_0002.json
│   │       └── episode_002/
│   │           ├── episode_content.json
│   │           ├── episode_summary.json
│   │           └── chunks/
│   │               └── chunk_0001.json
│   └── season_002/
│       ├── season_content.json
│       ├── season_summary.json
│       ├── character_stage_states.json
│       └── episodes/
└── character_cards/
    └── {character_id}.json
```

這種結構的好處是：

- 可以定位每條角色資訊來自哪一季、哪一集、哪一個 chunk。
- 中斷後可以從已完成的 chunk、集或季繼續。
- 角色卡生成時可以按時間順序讀取集級內容。
- 後續 UI 可以清楚展示角色成長來源。

## 7. 角色卡生成

角色卡生成採用步進式編譯：

```text
前一季角色狀態（可選）
-> 當前季第 1 集完整內容
-> 更新角色狀態
-> 當前季第 2 集完整內容
-> 更新角色狀態
-> ...
-> 當前季結束，生成階段總結
-> 下一季繼續
-> 最終整理
-> 輸出角色卡
```

這種方式更適合描述角色成長路線。它不會把角色看成一個靜態設定，而是把性格、關係、衝突和變化按時間逐步累積。

如果前後資訊出現矛盾，系統應記錄為角色的動態變化，例如偽裝、誤解、黑化、成長或關係轉折，而不是簡單覆蓋舊資訊。

## 8. 當前限制

第一階段不做複雜劇集識別，也不連網匹配番劇資料庫。

使用者需要提供相對合理的資料夾和檔案命名。系統先用簡單排序得到季和集的順序，之後可以在 UI 中增加手動調整順序。

這個設計故意保持透明：使用者能理解系統為什麼這樣排序，開發上也更容易保證可恢復、可追溯。
