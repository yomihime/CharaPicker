# 第三阶段：番剧、漫画、广播剧、小说等内容形态支持执行计划（zh_CN）

最近整理日期：2026-06-10

计划状态：正式执行计划，依赖路线 02 的架构图审核与媒体类型/内容形态接入准备完成。

适用阶段：路线 03，在路线 01 稳住提取质量与可观测性、路线 02 完成多媒体平级接入前重构解耦之后执行。若路线 02 尚未完成，只允许补充本计划、补充 02 的结构图和内容形态准备矩阵，不得直接实现路线 03 的提取功能。

执行闸门：开始路线 03 代码实现前，必须确认路线 02 已经由用户审核通过当前结构图、目标结构图和多媒体接入准备矩阵。该矩阵至少覆盖番剧、漫画、广播剧、小说、设定集、字幕/台本和混合资料包等内容形态，并逐项映射到 `video`、`image`、`audio`、`text` 四种代码层媒体类型，说明提取单元、预算口径、模型能力、证据引用、预览入口、正式入口、知识库落点和失败反馈。

## 1. 目的

CharaPicker 的长期方向是 Extract Once：不同来源的素材先被整理成可追溯的结构化知识库，后续角色卡、洞察和质量评估优先消费知识库，而不是反复分析原始素材。

当前稳定实现主要围绕番剧/视频类内容。视频预览、正式视频提取、视频 chunk 聚合、音频转写辅助和角色卡编译已经形成基础链路，但漫画、广播剧、小说、设定集、字幕/台本和混合资料包等内容形态仍没有完整进入预览、正式提取和知识库消费路径。

第三阶段的目标是补齐这些内容形态覆盖，让用户可以把番剧、漫画、广播剧、小说、设定集、台本、图片资料和混合资料包作为同一项目的素材来源，并让它们最终沉淀为同一套角色知识库事实。实现时不得把这些说法直接扩展成顶层代码枚举；代码层只使用 `video`、`image`、`audio`、`text` 四种媒体类型。

## 2. 范围

本计划覆盖多内容形态的真实接入，包括：

- 番剧/动画剧集：代码层主要为 `video`，可派生 `audio` 和 `text` transcript/subtitle。
- 漫画/图集：代码层为 `image`，页、页组、章节目录和阅读顺序属于内容形态 metadata。
- 广播剧/音频节目：代码层为 `audio`，可派生 `text` transcript。
- 小说/设定集/台本：代码层为 `text`。
- 字幕/歌词/台词稿：代码层为 `text`，可关联 `video` 或 `audio`。
- 图片资料、截图、插画：代码层为 `image`。
- 混合资料包：首版只支持 `video` + 字幕/台本类 `text` 的关联；同一目录中额外出现的图片、音频或文本会按媒体类型自动分入不同处理文件夹并独立管理，不在首版中与视频建立剧情关联。

已确认的首版默认：

- 文本类输出期望不新增用户设置，先由内部默认 helper 按文本 chunk 计算；不得复用视频每分钟口径。
- 内容形态先自动识别分类，不要求用户导入时手动选择；不同媒体类型和内容形态应进入不同处理文件夹或等价分类目录。
- 连续图片目录默认识别为漫画/图集候选，不要求用户手动标记“这是漫画”。
- 漫画/图集按文件夹分组：文件夹是首版章节、episode 或页组边界；不跨文件夹自动合并漫画页。
- 字幕/台词稿首批支持 `.srt` 和 `.ass`；`.vtt` 后置。
- GIF 首版不处理，不按静态图片、拆帧或动画理解进入提取链路。
- 混合资料包首版仅支持“视频 + 字幕/台本”；其他组合暂不支持，且同目录混放时只做分类管理，不做跨内容形态关联。
- 暂不新增受控测试素材；如后续需要新增，再同步素材来源声明。

本计划还覆盖：

- 这些素材在 `raw/`、`materials/`、`knowledge_base/` 中的路径和引用规则。
- 预览与正式提取如何共用素材单元、结构化产物和证据引用。
- 模型能力、预算口径、prompt、失败反馈、洞察流事件和 UI 状态。
- 真实素材验收、离线结构验证和用户数据安全规则。

## 3. 不做范围

- 不重新做路线 02 的架构重构。若路线 02 没有准备好通用扫描、素材单元、运行计划和模型能力分派，本阶段应先回补 02，而不是在 03 中绕过架构直接接素材。
- 不引入大而全的插件系统、动态第三方扩展市场或复杂 registry。素材处理器仍以项目内稳定边界为主。
- 不实现实时语音对话、语音生成、麦克风流式输入或语音角色扮演。
- 不把音频理解模型的自由回答当作逐字 transcript。正式对白事实优先来自 ASR/STT 转写。
- 不承诺所有模型供应商都支持图片、音频、视频音频联合理解或混合输入；能力必须逐 provider/schema/model 标记和验证。
- 不把抽帧图片模式描述成完整视频理解能力；抽帧默认没有音频信息。
- 不把预览产物默认作为正式知识库事实。
- 不让 GUI 层直接解析素材、拼接知识库路径或构建模型请求。
- 不绕过 `utils.ai_model_middleware` 直连模型后端。
- 不新增依赖，除非执行时证明没有标准库、现有工具或轻量实现可替代，并经用户确认。

## 4. 02 必须提前准备的事项

路线 02 不实现新内容形态提取，但必须为路线 03 做足结构准备。否则 03 会被迫在内容接入阶段再次做大规模架构移动。

02 的目标结构图和交接说明必须包含下列准备项：

| 准备项 | 02 要交付的最低要求 | 03 如何使用 |
| --- | --- | --- |
| 媒体类型 | `MediaType` 只能表达 `video`、`image`、`audio`、`text` | 03 不再新增 `subtitle`、`transcript`、`manga`、`mixed` 等平行顶层类型 |
| 内容形态 | `ContentForm` 或等价 metadata 能表达番剧、漫画、广播剧、小说、设定集、台本、混合资料包等说法 | 03 面向 UI、导入说明、提示词变量和用户反馈使用 |
| 提取单元 | `ExtractionUnit` 或等价结构不硬编码 `video_path`，能表达文本段、字幕段、音频片段、单图、漫画页组 | 03 为四种媒体类型填充 unit handler，并用内容形态补充语义 |
| 来源引用 | `MaterialRef` 使用相对 `materials/` 的路径和必要时间/页码/区域信息 | 03 生成可追溯 evidence |
| 证据引用 | `EvidenceRef` 至少能表达路径、时间范围、页码、区域、文本范围 | 角色卡编译和质量评估可追溯 |
| 预算入口 | 运行计划能区分输入硬预算与各媒体类型输出期望 | 03 接入番剧、漫画、广播剧、小说等内容形态时不复用错误口径 |
| 模型能力 | 能按媒体类型选择 text/image/audio/video 后端或返回可解释不支持 | 03 不在 UI 或处理器中硬编码 provider |
| 预览入口 | 预览能从通用素材单元采样，不只能收集视频 chunk | 文本、图片、漫画等可进入真实预览 |
| 正式入口 | full/clean/fast 能从通用运行计划读取素材单元 | 03 接入非视频正式提取 |
| 知识库落点 | chunk/episode/season/transcript 写入接口不绑定视频命名 | 03 写入统一知识库 |
| 失败反馈 | handler 能返回媒体类型、内容形态、来源路径、失败原因和是否可跳过 | UI/洞察流能解释某类内容不可用 |

执行路线 03 前必须检查这张矩阵。如果任意准备项缺失，应先开 02 补充阶段，不能直接在 03 里绕开通用边界。

## 5. 当前事实与差距

### 5.1 已有事实

- `utils.media_types` 已有 `VIDEO_SUFFIXES`、`IMAGE_SUFFIXES`、`TEXT_SUFFIXES`，`SUPPORTED_SOURCE_SUFFIXES` 当前覆盖视频、图片和文本。
- 音频后缀目前主要在 `utils.audio_transcription.AUDIO_SUFFIXES` 中定义，还没有进入通用素材导入后缀集合。
- `utils.source_importer` 和 `utils.source_status` 已围绕 `SUPPORTED_SOURCE_SUFFIXES` 处理 raw/materials 的导入、展示和状态。
- `utils.cloud_model_presets` 已有 text/image/video/audio backend 概念，以及 `audio_transcription`、`audio_understanding`、`video_audio_understanding` 等能力标记。
- `utils.ai_model_middleware` 是统一模型调用入口，现有视频和模型页测试能力不得被 GUI 绕过。
- `utils.audio_transcription` 可以从视频或音频素材生成 `EpisodeTranscript` 并保存 `episode_transcript.json`。
- `core.source_scanner` 的正式扫描和预览扫描仍以视频为核心。
- `core.extractor` 的预览和正式提取主入口仍以视频 chunk、视频输入模式和视频预算为主。
- `core.models.ExtractionStructuredArtifact`、`ChunkExtractionResult` 已有 `source_kind`、`source_counts`、`context_policy`、`token_usage`、`requested_output_tokens` 等可复用字段。
- `docs/plans/preview-real-result-ingestion-plan.zh_CN.md` 已保留小说/设定文本、字幕/台本、音频转写、图片/漫画进入预览和统一知识库消费路径的后续方向。

### 5.2 主要差距

- 小说/设定文本、字幕/台本、图片/漫画没有稳定的正式提取单元映射。
- 音频素材导入、音频转写、原生音频理解和视频内音频理解容易混在一起，需要明确能力层次。
- 预览入口当前最多处理前 2 个视频 chunk，不能从小说/设定文本、字幕/台本、图片或漫画生成 preview chunk。
- 正式提取入口仍从正式视频扫描计划开始。
- 知识库证据引用还不足以表达页码、图像区域、字幕时间轴、音频时间段和混合资料包来源。
- 预算与模型能力分派仍容易受视频每分钟语义影响。
- UI 尚未清楚区分“素材已导入”“素材已处理”“素材可提取”“模型不支持该内容形态/媒体类型”等状态。

## 6. 内容形态到媒体类型映射

### 6.1 小说、设定集与台本（代码层 text）

小说、设定集与台本等内容形态在代码层归入 `text`，包括 `.txt`、`.md` 和被明确声明为文本资料的 `.json`。

处理目标：

- 按章节、段落或固定 token/字符预算切分为文本单元。
- 保留源文件路径、标题、段落范围、字符范围和可选章节名。
- 通过文本模型生成与视频 chunk 一致语义的角色事实、关系、事件、冲突和不确定项。

注意：

- `.json` 不能无条件当作可信结构化知识库事实；默认先作为文本资料或受控导入格式处理。
- 文本原文可能很长，必须受输入预算硬约束。
- 输出 token 不得使用视频每分钟、图片每张或音频每分钟口径。

### 6.2 字幕、歌词与台词稿（代码层 text）

字幕、歌词与台词稿在代码层归入 `text`。首批只支持 `.srt` 和 `.ass`；`.vtt` 后置，不进入首版验收。

处理目标：

- 解析为带时间戳的字幕段。
- 与视频 episode 或独立 episode 对齐。
- 可作为 transcript 类文本证据进入知识库。
- 在存在视频时，可作为视频素材的补充上下文；在无视频时，也可独立形成文本提取单元。

注意：

- 字幕不是普通纯文本。证据引用应保留时间范围和原始字幕行号。
- 字幕不保证说话人准确；不得凭格式猜测角色身份。
- 如果字幕与视频时长或 episode 对齐失败，应给出明确 warning，而不是静默忽略。

### 6.3 广播剧/音频节目转写（代码层 audio -> text）

广播剧/音频节目转写指从 `audio` 或 `video` 音轨经 ASR/STT 生成 `text` transcript。当前优先路径是本地 `whisper.cpp`，可从视频音轨或音频文件生成 `EpisodeTranscript`。

处理目标：

- 把对白、旁白、可听文本沉淀为可追溯 transcript。
- 优先以 episode 级 `episode_transcript.json` 作为知识库落点。
- chunk 级只保存引用、时间范围或截取结果，不重复保存完整 transcript。

注意：

- Whisper 缺失不应阻塞不需要转写的素材流程。
- 音频转写失败要能让用户选择安装、跳过音频、切换为不需要 transcript 的模式，或停止运行。
- 日志不得记录完整转写文本。

### 6.4 广播剧/音频节目原生听觉理解（代码层 audio）

广播剧/音频节目的原生听觉理解指模型直接接收 `audio`，输出听觉摘要或结构化听觉线索。

处理目标：

- 补充语气、情绪、环境声、音乐、画外声音等 transcript 难以表达的信息。
- 作为独立 evidence 或 insight 进入知识库，而不是替代 transcript。

注意：

- 模型页音频测试通过不等于正式提取音频能力已接入。
- 原生音频输入必须按 provider/schema 走 `utils.ai_model_middleware`，UI 不拼请求体。
- 不支持音频的模型应返回可解释状态，不应把整次提取伪装成普通失败。

### 6.5 图片、截图与插画（代码层 image）

图片、截图与插画在代码层归入 `image`，包括单张图片、截图、插画和图片序列。

处理目标：

- 单图可成为一个 `image` unit。
- 图片序列可按目录、文件名排序或用户选择映射到 episode/unit。
- 模型请求应使用 image backend 或支持图片的多模态后端。
- 证据引用至少保存源路径和页/序号；后续可扩展图像区域。

注意：

- GIF 首版不处理，不按静态图片、拆帧或动画理解进入提取链路。
- 图片可能很大，需有尺寸、格式和失败原因反馈。
- 图片提取输出期望使用“期望图片输出 Token / 张”的软性口径。

### 6.6 漫画/图集（代码层 image）

漫画/图集在代码层归入 `image`，通常是有顺序的图片集合，可能按章节、页、跨页或目录组织。

处理目标：

- 按文件夹分组：文件夹是首版章节、episode 或页组边界，图片映射为该文件夹下的页 unit。
- 保留文件夹路径、页码、文件名排序、可选章节标题和页组关系。
- 首版优先使用视觉模型理解画面与文字，不强制引入 OCR 新依赖。
- 连续图片目录默认识别为漫画/图集候选，不要求用户手动标记“这是漫画”。

注意：

- 漫画页组不能只写“来自漫画”，必须有页码或文件路径证据。
- 读图顺序可能影响剧情理解，应允许后续校正排序规则。
- 不跨文件夹自动合并漫画页；跨文件夹章节关系留给后续手动整理或更高层 manifest。
- 大批量漫画页费用高，必须有 preview 限制、正式批处理进度和预算反馈。

### 6.7 混合资料包（首版仅 video + text 字幕/台本）

混合资料包指同一项目、season 或 episode 下存在多种内容形态，并由 `video`、`image`、`audio`、`text` 四种媒体类型组合表达。但路线 03 首版只支持番剧/视频与字幕/台本的关联，即 `video` + 时间轴 `text`。

处理目标：

- 用统一运行计划表达同一 episode 中的视频 unit 和字幕/台本 text unit。
- 同一事实可以同时引用视频 evidence 和字幕/台本 evidence。
- 同一目录中若还存在图片、音频或其它文本，先自动分入不同处理文件夹或等价分类目录，作为独立内容管理，不与该视频 episode 建立首版关联。
- 视频与字幕对齐失败时给出 warning 或跳过字幕关联，不得静默把字幕当作已对齐证据。

注意：

- 混合资料包不是 `mixed` 顶层媒体类型，也不等于把所有素材塞进同一次模型请求。
- 图片 + 文本、漫画 + 文本、音频 + 文本、视频 + 图片等组合不属于 03 首版支持范围。
- 首版重点是视频与字幕/台本证据对齐；更复杂的冲突合并留给后续阶段。

## 7. 数据与知识库规则

### 7.1 路径规则

保持现有项目根目录语义：

- `raw/` 保存可重新处理的源副本。
- `materials/` 保存当前处理管线实际消费的素材。
- `cache/` 保存临时处理、转码、转写和派生缓存。
- `knowledge_base/` 保存正式与预览结构化知识。
- `output/` 保存导出产物。

新增内容形态不得把用户素材写入 `res/`，不得把本地用户素材提交进仓库。

路线 03 首版不要求用户提供干净目录。导入或处理阶段应自动按媒体类型和内容形态分类到不同处理文件夹或等价分类目录；例如同一目录中同时存在视频和图片时，视频进入视频处理集合，图片进入图片/漫画处理集合，二者默认不建立剧情关联。旧视频项目的 flat `materials/` 读取路径必须保持兼容。

### 7.2 manifest 规则

路线 03 应复用路线 02 的通用 manifest 或 normalize 结果。

manifest 至少应能表达：

- `schema_version`
- `media_types`
- `content_forms`
- `source_root`
- season / episode / unit
- unit 的 `media_type`、`content_form`、`unit_kind`、`source_path`
- 时间范围、页码、图片区域、文本范围等可选 evidence metadata
- 派生素材关系，例如 transcript 派生自视频或音频

旧视频 manifest 必须继续可读。

### 7.3 提取产物规则

优先复用现有产物路径：

- `seasons/{season_id}/episodes/{episode_id}/chunks/{unit_id}.json`
- `episode_content.json`
- `episode_summary.json`
- `season_content.json`
- `season_summary.json`
- `episode_transcript.json`

新增字段应优先落在已有结构中：

- `source_kind` 或后续 `media_type`
- `content_form`
- `source_counts`
- `context_policy`
- `aggregation_warnings`
- `model_profile_id`
- `model_metadata`
- `token_usage`
- `estimated_context_tokens`
- `requested_output_tokens`
- evidence refs

如果 `ChunkExtractionResult` 不足以表达图片、漫画或字幕/台本证据，应先做兼容扩展，不应新增一套完全不同的知识库事实格式。

### 7.4 证据引用规则

证据引用必须能让用户和后续角色卡编译追溯来源。

建议最小字段：

| 内容形态/媒体类型 | 最小证据 |
| --- | --- |
| 小说/设定集/台本（`text`） | 文件相对路径、章节或段落、字符范围 |
| 字幕/歌词/台词稿（`text`） | 文件相对路径、字幕行号、开始/结束时间 |
| 转写文本（`audio`/`video` 派生 `text`） | `episode_transcript.json` 路径、segment index、开始/结束时间 |
| 广播剧/音频节目听觉理解（`audio`） | 音频相对路径、开始/结束时间、模型能力 |
| 图片/截图/插画（`image`） | 图片相对路径、序号、可选区域 |
| 漫画/图集（`image`） | 章节/episode、页码、图片相对路径、可选区域 |
| 视频 + 字幕/台本（`video` + `text`） | 视频 evidence refs、字幕/台本 evidence refs、对齐状态和对齐 warning |

## 8. 预算与模型能力

### 8.1 预算原则

- 输入/上下文窗口 Token 是硬约束。
- 输出 token 是软性期望，通过 prompt 或请求参数向模型表达，不保证模型严格遵守。
- 番剧/动画剧集中的 `video` 部分使用“期望视频输出 Token / 分钟”。
- 图片、截图、插画、漫画/图集单页等 `image` 部分使用“期望图片输出 Token / 张”。
- 广播剧/音频节目中的 `audio` 理解或音频摘要使用“期望音频输出 Token / 分钟”。
- 小说、设定集、台本、字幕/歌词/台词稿、transcript 等 `text` 部分不得偷偷复用视频、图片或音频口径。
- 角色卡编译使用“期望角色卡编译输出 Token”。

### 8.2 文本输出期望默认策略

路线 01 已定义输入预算、视频输出期望、图片输出期望、音频输出期望和角色卡编译输出期望，但没有单独定义“期望文本输出 Token / 块”。

路线 03 首版已确认不新增用户可见的文本类输出期望设置。小说/设定文本、字幕/台本和转写文本提取先由内部 helper 按文本 chunk 大小、上下文窗口和统一默认值计算软性输出期望，并在阶段报告中记录默认值来源。

该内部默认不得使用视频每分钟口径，也不得作为新的用户可见设置暴露。

### 8.3 模型能力分层

必须区分：

- `text`：处理小说、设定文本、台本、字幕、transcript、聚合、摘要。
- `image`：处理图片、截图、插画、漫画页、抽帧图片。
- `video`：处理原生视频或视频抽帧包装。
- `audio_transcription`：把音频转成 transcript。
- `audio_understanding`：理解音频中的非逐字听觉线索。
- `video_audio_understanding`：模型能同时理解视频画面和音轨。这里指模型 API 原生接收音视频输入并理解其内容，不是 Whisper/STT 转写，也不是抽帧图片理解。

模型能力不足时，应返回可解释状态。例如“当前模型不支持图片输入”“当前 API 不支持音频输入”“缺少 Whisper 运行时”“字幕解析失败”。

## 9. UI 与用户工作流

### 9.1 项目页

项目页需要支持用户导入更多内容形态，但仍只负责触发和反馈。

要求：

- 文件选择和文件夹导入应允许路线 03 支持的后缀。
- 素材列表应能显示面向用户的内容形态、代码层媒体类型、处理状态、是否可提取、是否需要额外工具。
- 处理按钮应能解释：复制/链接原素材、视频转码、音频转写、小说/设定文本解析、图片/漫画索引。
- 清理 raw 的安全规则不变。
- 页面层不解析字幕、不转写音频、不写知识库。

### 9.2 模型页

模型页需要帮助用户理解当前模型可处理哪些媒体类型，以及这些能力对应哪些内容形态。

要求：

- 文本、图片、音频、视频测试结果不应互相伪装。
- 音频测试仍是原生音频理解冒烟测试，不等于 Whisper/STT 转写测试。
- 不支持的媒体能力显示为“模型不支持”或“API 不支持”，并用用户能理解的内容形态解释影响，而不是普通失败。
- 需要 transcript 的流程应提示 Whisper 状态或可用转写后端。

### 9.3 洞察流

洞察流只展示用户关心的结构化事件。

要求：

- 事件 meta 包含 `media_type`、`content_form`、`unit_id`、`source_path`、`stage`。
- 不输出完整原文、完整 transcript、API Key 或大型素材内容。
- 跳过、失败、模型拒绝、预算截断和证据缺失应能区分。

## 10. Prompt 与结构化输出

新增内容形态需要独立 prompt surface 或清晰变量分支，但变量层仍应落到四种媒体类型和内容形态 metadata。

建议新增或扩展的目的：

- `preview_text_unit_extraction`
- `formal_text_unit_extraction`
- `preview_image_unit_extraction`
- `formal_image_unit_extraction`
- `formal_text_transcript_unit_extraction`
- `formal_audio_insight_extraction`
- `formal_image_page_sequence_extraction`
- `formal_multi_form_episode_merge`

输出结构应尽量与视频 chunk 保持一致：

- `facts`
- `behavior_traits`
- `dialogue_style`
- `relationship_interactions`
- `conflicts`
- `open_questions`
- `evidence`
- `quality_warnings`

Prompt 必须强调：

- 不臆造角色名和关系。
- 证据不足时写不确定项。
- 图片和漫画不能凭单页推断整个剧情。
- 音频理解不能替代逐字转写。
- 字幕说话人未知时不得强行归属。

## 11. 里程碑

### M01：执行前闸门与 02 就绪矩阵复核

目标：确认 02 已经为 03 准备好结构边界。

范围：

- 检查路线 02 当前结构图、目标结构图和用户审核记录。
- 检查 02 多媒体接入准备矩阵是否完整，并确认内容形态到四种媒体类型的映射没有缺口。
- 检查当前代码与 02 交接文档是否一致。

交付物：

- 03 执行前复核记录。
- 缺失准备项清单。
- 是否允许进入 03 代码实现的结论。

验收：

- 缺失项为零，或已明确先回补 02。
- 用户确认可以按 03 继续。

### M02：内容形态、媒体类型与后缀支持矩阵

目标：明确哪些内容形态可以导入、可以处理、可以预览、可以正式提取，以及它们对应哪一种代码层媒体类型。

范围：

- 更新或规划 `utils.media_types` 中 `video`、`image`、`audio`、`text` 四种媒体类型相关后缀。
- 明确 `.json`、漫画压缩包、未知扩展名的处理策略，并说明它们映射到哪种媒体类型和内容形态；`.gif` 首版不处理。
- 明确文件夹导入中图片序列、漫画目录、混合目录的自动识别规则；首版不要求用户手动选择内容形态。
- 明确处理后的不同媒体类型和内容形态进入不同文件夹或等价分类目录，同时保持旧视频 flat `materials/` 兼容。

验收：

- 支持矩阵能区分导入、预览、正式提取、需要额外工具、暂不支持。
- UI 文案不承诺未实现能力。

### M03：通用媒体扫描与内容形态 unit 映射落地

目标：从 `materials/` 生成多媒体 unit，并保留内容形态 metadata。

范围：

- 小说/设定文本映射为 `text` unit。
- 字幕/歌词/台词稿映射为带时间轴 evidence 的 `text` unit 或 transcript-like unit。
- 广播剧/音频节目映射为 `audio` unit，或映射为生成 `text` transcript 的派生任务。
- 图片、截图、插画映射为 `image` unit。
- 连续图片目录默认映射为漫画/图集候选，并生成带页码/页组 metadata 的 `image` unit。
- 混合目录先按媒体类型拆分到不同分类集合；只有视频 + 字幕/台本会尝试 episode 关联，其它组合独立管理。

验收：

- 不同媒体类型和内容形态都能进入同一运行计划。
- 旧视频扫描兼容。
- unit 有稳定排序、来源路径和 evidence metadata。

### M04：小说、设定集与台本解析和 chunking

目标：让小说、设定集、台本等 `text` 内容进入预览和正式提取。

范围：

- 读取 `.txt`、`.md` 和受控 `.json`。
- 按输入预算切分文本 chunk。
- 建立文本 evidence。
- 调用 text backend 生成结构化结果。

验收：

- 无视频但有文本类内容时，预览能生成 preview chunk。
- 正式提取能写入 full chunk、episode content 和 summary。
- 超长文本会截断或分块并给出 warning。

### M05：字幕、歌词与台词稿接入

目标：让字幕、歌词与台词稿成为可追溯 `text` 内容。

范围：

- 解析 `.srt`、`.ass`。
- `.vtt` 首版提示暂不支持，后续再接入。
- 保留时间轴、行号和原始文本。
- 能与视频 episode 关联，也能独立形成文本提取单元。

验收：

- 字幕预览和正式提取可运行。
- 证据引用包含时间范围。
- 说话人未知时不会强行归属。

### M06：广播剧/音频节目转写作为正式证据入口

目标：把广播剧、音频节目或视频音轨转写结果作为知识库事实来源。

范围：

- 支持音频文件进入素材导入和转写任务。
- 复用 `utils.audio_transcription` 写入 `episode_transcript.json`。
- 从 transcript 构建提取 unit。
- 转写失败给出可解释反馈。

验收：

- 音频文件可生成 transcript。
- transcript 可进入预览和正式提取。
- Whisper 缺失只影响需要转写的流程。

### M07：原生音频理解接入

目标：补充 transcript 不覆盖的听觉线索。

范围：

- 按模型能力判断是否可直接音频输入。
- 构建音频理解请求。
- 保存听觉摘要、语气、环境声等 evidence。

验收：

- 支持音频的模型可返回结构化听觉线索。
- 不支持音频的模型有明确状态。
- 音频理解不覆盖 transcript 事实。

### M08：图片、截图与插画接入

目标：让单图、截图、插画和图片序列进入预览和正式提取。

范围：

- 图片 unit 构建。
- image backend 请求。
- 图片证据路径、序号和可选区域。
- 图片输出期望使用“每张”软性口径。

验收：

- 无视频但有图片时，预览可生成 preview chunk。
- 正式提取可写 full chunk。
- 图片过大、格式不支持或模型不支持时反馈明确。

### M09：漫画/图集接入

目标：让漫画目录或页组以 `image` unit 进入知识库。

范围：

- 漫画目录识别与排序。
- 按文件夹分组，建立文件夹、页、章节和 episode 映射。
- 漫画页视觉提取。
- 页组聚合为 episode content。

验收：

- 漫画文件夹可生成稳定 units，且不会跨文件夹自动合并。
- 证据引用包含页码或文件名。
- 预览限制成本，正式提取显示批处理进度。

### M10：视频 + 字幕/台本 episode 对齐

目标：让番剧/视频与字幕/台本在同一 episode 中形成可追溯的首版混合证据。

范围：

- 扫描同一目录或同一 episode 下的视频与字幕/台本文件。
- 建立视频 unit 与字幕/台本 `text` unit 的关联 metadata。
- 对齐失败时记录 warning，并允许仅按视频或仅按字幕继续。
- 同一目录中出现的图片、音频或其它文本分入各自分类集合，不与视频 episode 建立首版关联。

验收：

- 番剧/视频 + 字幕/台本组合可表达。
- 聚合结果保留视频和字幕/台本两类证据来源。
- 图片 + 设定文本、漫画 + 小说/设定文本、音频 + 文本等组合不会被误标为首版已支持。

### M11：预览链路多内容形态化

目标：预览不再只依赖视频 chunk。

范围：

- 从通用 unit 中采样预览内容。
- 保持 preview artifact 隔离。
- 针对不同媒体类型和内容形态显示合适进度和 warning。

验收：

- 没有视频时，小说/设定文本、字幕/台本、图片或漫画仍能预览。
- 预览不消费正式产物。
- 预览产物不进入正式角色卡编译。

### M12：正式提取多内容形态化

目标：正式 full/clean/fast 能消费多媒体 unit。

范围：

- full 路径按媒体单元提取并聚合。
- clean 路径遵守正式产物重建语义。
- fast 路径只在媒体处理器支持并发时启用。
- 不支持并发的媒体类型回退为串行并说明。

验收：

- 视频旧路径不回退。
- 小说/设定文本、字幕/台本、transcript、图片至少有正式提取路径。
- 漫画/图集和视频 + 字幕/台本混合资料包按本阶段完成范围通过验收。

### M13：知识库证据与角色卡消费

目标：让角色卡编译能消费多内容形态的正式知识库。

范围：

- 统一 `source_kind`、`media_type`、`content_form`、`source_counts` 和 evidence。
- 角色卡编译不回头分析原始素材。
- 质量评估可识别不同媒体类型和内容形态证据的可靠性。

验收：

- 角色卡编译能从多内容形态知识库读到事实。
- evidence 不丢失媒体类型、内容形态和来源。
- 旧视频项目仍可编译。

### M14：模型拒绝与失败样例记录适配

目标：让多内容形态失败和模型拒绝能进入 01 设计的拒绝样例流程。

范围：

- 失败记录包含 `media_type`、`content_form`、unit、source path、模型、prompt purpose、错误类型。
- 用户打包拒绝样例时，能复制相关素材和 JSON。
- 大型素材按索引引用或受控复制，不泄露无关项目文件。

验收：

- 图片、音频、文本、漫画等内容形态失败都能生成可回传的结构化记录。
- 记录不包含 API Key 或完整隐私文本。

### M15：GUI 与 i18n

目标：让用户看得懂多内容形态状态。

范围：

- 内容形态与媒体类型显示。
- 处理状态和失败反馈。
- 模型能力不支持提示。
- 预算设置说明。
- 四个 i18n JSON 同步。

验收：

- UI 不把未支持内容形态显示为可正式提取。
- 所有新增文案走 i18n。
- 长耗时任务仍在 worker/thread。

### M16：离线验证与测试素材

目标：补足不依赖云端模型的结构验证。

范围：

- 扫描和 unit 映射测试。
- 文本 chunking 测试。
- 字幕解析测试。
- transcript 读取测试。
- 图片/漫画排序测试。
- manifest 兼容测试。
- 暂不新增受控固定测试素材；需要结构样例时优先在测试临时目录中构造最小文本、字幕或图片占位，不把新素材文件提交进仓库。

验收：

- 不需要真实模型即可验证核心结构。
- 若后续确需新增固定测试素材，必须先同步 `docs/reference/asset-material-declaration.zh_CN.md`。

### M17：真实模型和手动验收

目标：用真实素材确认多内容形态链路可用。

范围：

- 小说/设定文本项目。
- 字幕/台本项目。
- 广播剧/音频转写项目。
- 图片/插画项目。
- 漫画目录。
- 至少一个视频 + 字幕/台本项目。

验收：

- 预览和正式提取都能运行。
- 知识库产物可检查。

- 文件夹导入和 raw/materials 映射。
- manifest normalize。
- unit 排序和 evidence。
- 文本 chunking。
- 字幕解析。
- transcript 缓存读取。
- 图片/漫画排序。
- preview/full artifact 隔离。

### 13.3 手动验收

至少准备：

- 一个小说/设定文本项目。
- 一个字幕/台本项目。
- 一个广播剧/音频或视频音轨转写项目。
- 一个单图/插画项目。
- 一个漫画目录项目。
- 一个视频 + 字幕/台本项目。
- 一个模型不支持目标内容形态的失败项目。

## 14. 风险与缓解

| 风险 | 表现 | 缓解 |
| --- | --- | --- |
| 02 准备不足 | 03 又开始移动架构 | 先回补 02 就绪矩阵 |
| 模型能力误判 | UI 显示可用但请求失败 | provider/schema/model 分层标记 |
| 预算口径混乱 | 文本类内容误用视频每分钟 | 按媒体类型分派预算 |
| 音频概念混淆 | 音频理解替代 transcript | ASR 与音频理解分开 |
| 漫画费用过高 | 大量页面一次性发送 | 页组、采样、进度和预算提示 |
| 证据不可追溯 | 角色卡事实找不到来源 | EvidenceRef 强制路径/时间/页码 |
| 预览污染正式 | preview 产物被正式编译消费 | 继续使用 preview stage 和前缀隔离 |
| UI 越界 | 页面直接解析素材或调模型 | worker/core/utils 边界审查 |
| 隐私泄露 | 日志记录原文或 transcript | 只记录摘要和路径，不记录完整内容 |

## 15. 执行时待确认事项

- 原生音频/视频理解首批是否真的调用支持音频或音视频输入的模型，还是只保留模型能力判断和“暂不支持”反馈。

## 16. 代码审查自检表

执行 03 的每个阶段提交前，至少自查：

- 是否已经确认 02 的结构图和多媒体接入准备矩阵通过用户审核。
- 是否绕过了路线 02 的通用媒体单元、运行计划或模型能力分派。
- 是否新增了 UI 直接解析素材、写知识库或拼模型请求的行为。
- 是否绕过 `utils.ai_model_middleware`。
- 是否把音频理解当作 transcript。
- 是否把预览产物当正式知识库事实。
- 是否把视频每分钟预算用于文本、图片或音频之外的场景。
- 是否新增了用户可见的文本类输出期望设置；首版应使用内部默认 helper。
- 是否要求用户手动标记漫画；首版连续图片目录应自动识别为漫画/图集候选。

- 是否有日志输出完整原文、完整 transcript、API Key 或隐私内容。

## 17. 完成定义

路线 03 完成后，应满足：

- 小说/设定文本、字幕/台本、音频转写、图片、漫画和视频 + 字幕/台本组合可以进入预览。
- 小说/设定文本、字幕/台本、transcript、图片至少可以进入正式提取并写入知识库。
- 漫画/图集具备稳定 `image` unit 映射、证据引用和至少首版正式提取路径。
- 同一目录中混放的视频、图片、音频和文本能自动分类进不同处理集合；除视频 + 字幕/台本外，不建立首版跨内容形态关联。
- 音频理解能力与音频转写能力分层明确。
- 角色卡编译能消费多内容形态正式知识库。
- 模型不支持或工具缺失时，用户能看到明确反馈。
- 旧视频预览、正式提取和角色卡编译不回退。
- 架构文档、工作流文档和 TODO 状态已同步。
