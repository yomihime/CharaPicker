# 文档维护规范（zh_CN）

本文记录 CharaPicker 的长期文档维护规则。它用于替代旧的临时 AI 上下文，成为人类与 AI 都可查阅的正式文档规范。

## 1. 文档入口

- 根目录 `README.md` 是 GitHub 首页说明，默认使用简体中文。
- 多语言 README 放在 `docs/`，文件名带语种后缀，例如 `README.en_US.md`。
- 设计说明、开发路线、发布说明和专项计划统一放在 `docs/`。
- `AGENTS.md` 是 Codex 默认自动加载的长期项目指导文件，不记录临时任务进度。
- 后续工作流不依赖 `.codex/`；`.codex/CONTENT.md` 只能作为历史迁移来源，不作为正式文档入口。

## 2. README 规范

根目录 `README.md` 应保持简洁，面向首次访问仓库的人说明：

- 项目定位。
- 当前状态。
- 已实现与未完成内容。
- 环境要求。
- 安装、运行、构建方式。
- 文档导航。

不要把 README 改成完整开发手册；详细规范应放入 `docs/` 或对应 `ARCHITECTURE.md`。

## 3. 多语言文档规范

- 多语言文档文件名必须能区分语种。
- 当前基础语种为简体中文、繁体中文、英语和日语。
- 简体中文是默认维护语言；其他语言文档可以作为补充版本存在。
- 修改多语言用户文档时，应注意不同语言版本是否需要同步。

## 4. ARCHITECTURE.md 规范

每个主要目录应有 `ARCHITECTURE.md`。根目录也必须有架构说明，并链接到各主要目录的架构说明。

每份架构说明至少包含：

- 该目录负责什么。
- 该目录不负责什么。
- 关键文件说明。
- 与其他目录的数据流或调用关系。
- 后续维护注意事项。

写作要求：

- 面向人和 AI 同时可读。
- 句子短，边界清楚。
- 少用隐喻，优先说明职责和禁止事项。
- 不把路线图写成当前事实。

## 5. 计划类文档规范

roadmap、专项计划和设计草案可以保留，但必须避免被误读为已实现事实。

计划类文档应尽量说明：

- 当前问题。
- 接入目标。
- 实现顺序。
- 模块边界。
- 验收标准。
- 文档是否可能过期，以及实施前应核对哪些代码。

如果计划已经明显被代码推进或替代，应在文档顶部补充状态提醒，而不是静默留下过时描述。

## 6. 文档更新时机

出现以下变化时，应考虑更新正式文档：

- 模块职责发生变化。
- 新增主要目录或稳定中间件。
- 项目数据结构或知识库结构发生变化。
- 构建、发布、版本规则发生变化。
- 用户可见流程或核心业务语义发生变化。
- 新增长期有效的 Codex 工作约束。

如果变化只属于单个临时任务，不应写入 `AGENTS.md`；优先写入专项文档或任务说明。
