# 文档维护规范（zh_CN）

本文记录 CharaPicker 的长期文档维护规则。它用于替代旧的临时 AI 上下文，成为人类与 AI 都可查阅的正式文档规范。

## 1. 文档入口

- 根目录 `README.md` 是 GitHub 首页说明，默认使用简体中文。
- 多语言 README 放在 `docs/readme/`，文件名带语种后缀，例如 `README.en_US.md`。
- 设计说明、开发路线、发布说明和专项计划统一放在 `docs/`，数量较多时按职责放入 `reference/`、`plans/` 或 `archive/`。
- `AGENTS.md` 是 Codex 默认自动加载的长期项目指导文件，不记录临时任务进度。
- 后续工作流不依赖 `.codex/`；`.codex/CONTENT.md` 只能作为历史迁移来源，不作为正式文档入口。

## 2. README 规范

根目录 `README.md` 应保持简洁，面向首次访问仓库的人说明：

- 项目定位。
- 当前状态。
- 已实现与未完成内容。
- 环境要求。
- 安装、运行、构建方式。
- 文档导航。

不要把 README 改成完整开发手册；详细规范应放入 `docs/` 或对应 `ARCHITECTURE.md`。

## 3. 多语言文档规范

- 多语言文档文件名必须能区分语种。
- 当前基础语种为简体中文、繁体中文、英语和日语。
- 简体中文是默认维护语言；其他语言文档可以作为补充版本存在。
- 修改多语言用户文档时，应注意不同语言版本是否需要同步。

## 4. ARCHITECTURE.md 规范

每个主要目录应有 `ARCHITECTURE.md`。根目录也必须有架构说明，并链接到各主要目录的架构说明。

每份架构说明至少包含：

- 该目录负责什么。
- 该目录不负责什么。
- 关键文件说明。
- 与其他目录的数据流或调用关系。
- 后续维护注意事项。

写作要求：

- 面向人和 AI 同时可读。
- 句子短，边界清楚。
- 少用隐喻，优先说明职责和禁止事项。
- 不把路线图写成当前事实。

## 5. 计划类文档规范

roadmap、专项计划和设计草案可以保留，但必须避免被误读为已实现事实。

计划类文档应尽量说明：

- 计划阶段，例如当前队列、待执行专项、长期路线基准、已完成归档。
- 可用性，例如可直接执行、实施前需复核代码、仅作历史参考。
- 当前问题。
- 接入目标。
- 实现顺序。
- 模块边界。
- 验收标准。
- 文档是否可能过期，以及实施前应核对哪些代码。

如果计划已经明显被代码推进或替代，应在文档顶部补充状态提醒，而不是静默留下过时描述。

如果计划已经执行完成，或只剩历史查阅价值，应移入 `docs/archive/`，并在 `docs/README.md` 和 `docs/ARCHITECTURE.md` 中保持入口说明，避免主文档目录被一次性执行记录占满。

已完成计划中仍有独立未完成项时，应先把未完成项整理进 `docs/plans/TODO.zh_CN.md`，再归档原计划。若没有独立残项，可以在归档文档顶部写明归档复核日期和“未发现需迁入 TODO 的独立未完成项”。

当前目录分组约定：

- `docs/readme/`：README 的多语言补充版本。
- `docs/reference/`：长期参考材料、声明和不直接驱动排期的规范文档。
- `docs/plans/`：仍需执行或持续跟踪的任务队列和专项计划。
- `docs/archive/`：已完成、已替代或主要保留历史查阅价值的计划。

## 6. reference 文档规范

`docs/reference/` 容易因为长期材料和多语言补充堆叠而变得难读，维护时优先整理导航和状态，而不是频繁移动稳定路径。

- `docs/reference/README.md` 必须按阅读目的分组，而不是只按文件名罗列。
- 简体中文文档默认是维护主版本；多语言参考文档需要在索引中标明其补充属性。
- 当前代码和架构文档能直接确认的事实，不要在 reference 中重复成大段长期噪音。
- 已经变成执行队列的内容移入 `docs/plans/`；已经变成历史记录的内容移入 `docs/archive/`。
- 移动 reference 文件前先检查 `AGENTS.md`、`docs/README.md`、`docs/ARCHITECTURE.md` 和相关架构文档中的稳定链接。

## 7. 文档更新时机

出现以下变化时，应考虑更新正式文档：

- 模块职责发生变化。
- 新增主要目录或稳定中间件。
- 项目数据结构或知识库结构发生变化。
- 构建、发布、版本规则发生变化。
- 用户可见流程或核心业务语义发生变化。
- 新增长期有效的 Codex 工作约束。

如果变化只属于单个临时任务，不应写入 `AGENTS.md`；优先写入专项文档或任务说明。
