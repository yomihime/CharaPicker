# 抽出ワークフロー技術説明（ja_JP）

この文書は、ユーザー、研究者、または CharaPicker の設計を理解したい人向けに、長編動画素材がどのように抽出、圧縮、整理され、最終的にキャラクターカード生成に使われるかを説明します。

## 1. 設計目標

CharaPicker の基本原則は `Extract Once` です。元の動画、字幕、画像素材は一度だけ解析し、その後は再利用可能な構造化ナレッジとして保存します。

このワークフローは次の問題を解決します。

- 長編アニメや動画素材は長すぎて、モデルへ一度に入力できない。
- キャラクターの成長には時系列が必要で、孤立した断片の要約だけでは足りない。
- 後続のキャラクターカード生成では、元動画を再解析するのではなく、構造化結果を優先して読むべきである。

そのため、システムは素材を「シーズン、エピソード、chunk」の三層に分けます。

- `chunk` は抽出段階の処理単位で、モデルのコンテキスト長を制御するために使います。
- `エピソード` は物語理解とキャラクター成長の最小の自然単位です。
- `シーズン` は段階的な成長、関係変化、長期的な衝突をまとめる単位です。

## 2. 入力素材の規約

第一段階では、単純で説明しやすいディレクトリ認識ルールを使います。

- ユーザーは素材のルートフォルダーを選択します。
- ルート直下の各フォルダーを 1 シーズンとして扱います。
- シーズンフォルダー内の動画ファイルを、そのシーズンのエピソードとして扱います。
- シーズンフォルダーとエピソードファイルは、デフォルトで名前順に並べます。

推奨される命名：

- シーズンフォルダー：`01 To LOVEる`、`02 Motto To LOVEる`、`03 Darkness`
- エピソードファイル：`01 xxx.mp4`、`02 xxx.mp4`、`10 xxx.mp4`
- `xxx S01`、`xxx S02` のような名前も使えますが、テキスト順で実際の順序になる必要があります。

`01`、`02`、`10` のようなゼロ埋め番号を推奨します。これにより、単純なテキストソートでも正しい順序を得られます。

インポート後、システムは `source_manifest.json` を生成し、元のフォルダー名、元のファイル名、内部 ID の対応を記録します。以降の処理では `season_001`、`episode_001`、`chunk_0001` のような安定した ID を使い、元ファイル名から繰り返し推測しません。

## 3. 全体フロー

推奨フロー：

```text
元素材
-> フォルダーからシーズンを識別
-> ファイル名順でエピソードを識別
-> 各エピソードを chunk に分割
-> 各 chunk から構造化結果を抽出
-> エピソード単位の完全内容へ統合
-> エピソード単位の圧縮要約を生成
-> シーズン単位の完全内容へ統合
-> シーズン単位の圧縮要約を生成
-> シーズン、エピソード順にキャラクター状態を段階的に編纂
-> 最終キャラクターカードを生成
```

重要な設計判断として、キャラクターカード生成は chunk から始めません。

chunk は、モデルが長い素材を処理できるようにするための単位です。キャラクター成長を再現する段階では、エピソード単位の完全内容から始め、エピソードごとに進めます。エピソードは chunk よりも物語構造に合っており、キャラクター変化を観察する単位として自然です。

## 4. 抽出時のコンテキスト

現在の chunk を抽出するとき、システムは次の優先順位でコンテキストを組み立てます。

1. 現在の chunk 内容。
2. 現在のエピソードで完了済み chunk の完全な構造化抽出結果。
3. 現在のシーズンで完了済みエピソードのエピソード要約。
4. 前シーズンのシーズン要約。

現在の chunk は常に最優先の証拠です。

同じエピソードは通常それほど長くないため、同一エピソード内で抽出済みの chunk は短い要約ではなく「完全な構造化結果」として渡せます。ただし、ここでの「完全」とは構造化抽出結果のことであり、元字幕や原文全文ではありません。これにより、詳細を保ちながら、処理済みの原文でコンテキストを浪費することを避けられます。

非常に長いエピソードの場合は、直近の chunk の完全な構造化結果を残し、それ以前の chunk をエピソード内ローリング要約へ統合できます。

## 5. エピソード間とシーズン間

同じシーズン内では、後続エピソードに前の完了済みエピソードの圧縮要約を渡します。これにより、モデルは関係性、衝突、キャラクター状態の連続性を保ちやすくなります。

シーズンをまたぐ場合、前シーズンのシーズン要約を渡せますが、低優先度の背景として扱います。これは現在のシーズンに入る前のキャラクター状態、関係性、未解決の衝突を説明するためのもので、現在シーズンの素材に含まれる新事実を上書きしてはいけません。

前シーズン要約は意味上、次のようにラベル付けします。

```text
PREVIOUS_SEASON_BACKGROUND
```

つまり、前シーズン情報は背景であり、現在の証拠ではありません。

## 6. ナレッジベース構造

抽出結果はプロジェクトの `knowledge_base` に書き込まれ、シーズン、エピソード、chunk の階層で保存されます。

推奨構造：

```text
knowledge_base/
├── source_manifest.json
├── seasons/
│   ├── season_001/
│   │   ├── season_content.json
│   │   ├── season_summary.json
│   │   ├── character_stage_states.json
│   │   └── episodes/
│   │       ├── episode_001/
│   │       │   ├── episode_content.json
│   │       │   ├── episode_summary.json
│   │       │   └── chunks/
│   │       │       ├── chunk_0001.json
│   │       │       └── chunk_0002.json
│   │       └── episode_002/
│   │           ├── episode_content.json
│   │           ├── episode_summary.json
│   │           └── chunks/
│   │               └── chunk_0001.json
│   └── season_002/
│       ├── season_content.json
│       ├── season_summary.json
│       ├── character_stage_states.json
│       └── episodes/
└── character_cards/
    └── {character_id}.json
```

この構造には次の利点があります。

- 各キャラクター情報が、どのシーズン、どのエピソード、どの chunk に由来するか追跡できる。
- 中断後、完了済みの chunk、エピソード、シーズンから再開できる。
- キャラクターカード生成時に、エピソード単位の内容を時系列で読める。
- 後続の UI で、キャラクター成長の根拠を明確に表示できる。

## 7. キャラクターカード生成

キャラクターカード生成は段階的な編纂を使います。

```text
前シーズンのキャラクター状態（任意）
-> 現在シーズン第 1 話の完全内容
-> キャラクター状態を更新
-> 現在シーズン第 2 話の完全内容
-> キャラクター状態を更新
-> ...
-> 現在シーズン終了、段階要約を生成
-> 次シーズンへ継続
-> 最終整理
-> キャラクターカードを出力
```

この方式は、キャラクターの成長経路を表現するのに向いています。キャラクターを静的な設定として扱うのではなく、性格、関係、衝突、変化を時間に沿って積み重ねます。

前後の情報が矛盾する場合、システムはそれを単純に上書きせず、偽装、誤解、闇落ち、成長、関係の転換などの動的変化として記録します。

## 8. 現在の制限

第一段階では、複雑なエピソード識別やアニメデータベースとのオンライン照合は行いません。

ユーザーは、ある程度整理されたフォルダー名とファイル名を用意する必要があります。システムはまず単純なソートでシーズンとエピソードの順序を決め、後から UI で手動調整を追加できます。

この設計は意図的に透明性を保っています。ユーザーはシステムがなぜその順序にしたか理解しやすく、実装面でも再開と追跡がしやすくなります。
